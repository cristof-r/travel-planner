# Travel Planner — Obsidian Plugin

TypeScript Obsidian plugin. Packing lists are the first feature; itinerary/bookings/budget are planned but not built. Node 18+.

## Commands

- `pnpm dev` — watch build (development)
- `pnpm build` — production build
- `pnpm lint` — ESLint on src/
- `pnpm test` — run vitest tests
- `.\build.ps1 -VaultDir "C:\path\to\vault"` — deploy to vault (Windows)
- `./build.sh "/path/to/vault"` — deploy to vault (Linux/macOS)

Deploy copies `main.js` and `manifest.json` into `.obsidian/plugins/travel-planner/`.

## Structure

```
src/
  main.ts            # Plugin entry, commands, modals, trip-folder path helpers
  packing-list.ts    # Pure logic: trip-item/Base generation, catalog parsing, category color/migration (unit tested)
  settings.ts         # Settings tab + CategoryEditModal (name/color popup): trips folder, catalog folder, categories, typed packing columns
  constants.ts        # Default category names, example catalog data, Trips Overview Base template
  types.ts             # Shared TypeScript types (CategoryConfig, PackingColumn, PackingCatalogItem, PackingListSettings)
  i18n/
    index.ts          # t() translation function + locale detection
    en.ts              # English strings (source of truth for translation keys)
    de.ts               # German strings
  ui.ts                # Shared DOM helpers (appendCategoryDot) used by main.ts and settings.ts
docs/
  requirements.md              # Full app requirements + market research
  packing-list/requirements.md # Packing-list-specific requirements (current focus)
```

## Data & Note Conventions

- **A trip is a folder (`Trips/<name>/`), not a single note** — chosen specifically so the packing list can be a *real* interactive table: Markdown tables can't hold clickable checkboxes (only a single leading `- [ ]` on a list line is ever interactive; anything inside a `|` cell is inert text), but Obsidian Bases renders boolean frontmatter properties as genuinely clickable checkboxes. The folder holds:
  - `<name>.md` — the trip note (`trip: true`, `destination`, `start_date`, `end_date` frontmatter), with the packing table embedded via `![[<tripDir>/Packing.base]]` (full vault-relative path, since every trip's Base is named identically — a bare `![[Packing.base]]` would be ambiguous).
  - `items/<item>.md` — one small note per packed item: its category (`buildTripItemContent()`), a `quantity` (always present, default `1`, not part of the configurable columns), and one entry per configured packing column at its type's zero value (`0` for `"number"`, `false` for `"boolean"`).
  - `Packing.base` — generated by `buildTripPackingBase()`, filtering `file.folder == "<items dir>"`, with category, quantity, and every packing column as real, typed table columns.
- `tripPaths()` in `main.ts` derives all three paths from a trips-root + trip name; `createTripItemNotes()`/`createTripBase()` are the shared, idempotent (skip-if-exists) writers used by `createTrip`, `insertPackingList`, and `createExampleData`.
- `insertPackingList`/`resetCurrentPackingList` operate relative to the **active file's own parent folder** (`file.parent.path`) rather than a trip name, so they work whether the active note is a `createTrip`-generated trip note or one you made yourself.
- `resetCurrentPackingList` uses `app.fileManager.processFrontMatter()` (not text/regex editing) to set every configured packing column back to its type's zero value (`0` or `false`) on every note in the sibling `items/` folder — resets *all* columns, not just a hardcoded "packed" one, since columns are user-configurable and there's no reliable way to guess which one means "packed" for a custom column set. `quantity` is never reset — it represents "how many I need," not packing progress.
- **Item catalog is read live from vault notes**, not plugin data, using **fixed, non-configurable** frontmatter field names: `CATALOG_NAME_FIELD`/`CATALOG_CATEGORY_FIELD` (`constants.ts`) = `travel-planner-name`/`travel-planner-category`, namespaced so they don't collide with other plugins' frontmatter on the same notes. The catalog folder itself (default `Trips/Packing Catalog`) is still a setting. `getCatalog()` in `main.ts` reads the folder via the metadata cache on every command invocation — there is no cached/stored copy to go stale. `isCatalogConfigured()` now only checks `catalogFolder` truthiness (field names can't be cleared, since they're not settings); `getCatalog()` returns `[]` when unconfigured, so packing lists degrade to empty rather than erroring. Only `categories` (display order + color), `packingColumns`, and `catalogFolder`/`tripsFolder` live in plugin data.
- **`PackingCatalogItem.categories` is a list, not a single string.** The category frontmatter field may be a bare scalar or a YAML list; `parseCatalogItems`'s internal `normalizeCategories()` accepts either so single-category notes keep working unchanged. `formatCategoriesFrontmatter()` writes a bare scalar back out for one category and a list for several, so the plugin never introduces a single-item list where a scalar would do. `parseCategoriesInput()` parses the comma-separated free text typed into the "Create Packing Item Note" category prompt. Everywhere an item is matched against a category (`mergeCategories`, `buildCatalogBase`, the trip-item filter in `createTrip`/`insertPackingList`), it checks list membership, not equality — an item with several categories is included whenever any of its categories is selected.
- **`settings.categories` is `CategoryConfig[]` (`{ name, color }`), not `string[]`.** Colors are hex strings, editable via Obsidian's native `ColorComponent`/`addColorPicker()`. `categoryColor()` (`packing-list.ts`) computes a deterministic default (name → hash → hue → hex) only for *new* categories — one just discovered in the catalog via `mergeCategories()`, or one added through the settings "Add category" field; an existing entry's color is never recomputed or overwritten, so a user's manual color choice sticks. `appendCategoryDot()` (`ui.ts`) just renders a given color — it doesn't compute one — so every call site (settings list, `CategoryPickerModal`, `CommaListSuggest`) passes the category's real configured color, not a fallback.
- **Renaming a category in Settings (`CategoryEditModal`) does not rename it inside existing item notes.** The settings list is a display-order + color registry, decoupled from the frontmatter values actually written in item notes. After a rename, old items with the previous name reappear via `mergeCategories()` as a new, uncolored-by-you entry appended at the end — a known, accepted limitation, not a bug.
- **`settings.packingColumns` is `PackingColumn[]` (`{ key, label, type }`)**, `type` being `"boolean" | "number"`. Default: `[{ key: "packed", label: "Packed", type: "number" }, { key: "bought", label: "Bought", type: "boolean" }]` — Packed is a number so it can track a *packed count* against `quantity`, Bought stays a plain checkbox. `key` is derived from the label via `slugifyPropertyKey()` when a column is added in Settings (type is chosen via a dropdown at add-time and can't be changed afterward — remove and re-add instead), and used verbatim as the frontmatter property name in every trip-item note and the generated `.base` file. Bases infers checkbox-vs-number-input rendering from the property's actual value type, so no extra Bases-syntax work is needed beyond writing the right zero value.
- **Settings migration**: pre-color-picker versions of the plugin stored `categories` as a plain `string[]`. `loadSettings()` runs `migrateCategoryConfigs()` (`packing-list.ts`) on the raw loaded data before merging with `DEFAULT_SETTINGS`, upgrading old string entries to `{ name, color: categoryColor(name) }` in place. Keep this migration until it's safe to assume no installs predate it. There is deliberately **no** migration for pre-Bases-table trip notes (the old single-note-with-inline-checklist format) — those are left untouched; `resetCurrentPackingList` on one of them just reports no packing list found, since there's no `items/` folder to reset.
- **Overview `.base` files** (`buildCatalogBase()`/`buildTripPackingBase()` in `packing-list.ts`, `TRIPS_BASE_CONTENT` in `constants.ts`) are generated once, then owned entirely by the user — the plugin never rewrites an existing `.base` file, only refuses to overwrite one that's already there (see `notice.*OverviewExists`, and the exists-checks in `createTripBase`/`createTripItemNotes`).
- **`createExampleData()` is the one exception to "no defaults."** Every other command refuses via `isTripsConfigured()`/`isCatalogConfigured()` when settings are empty; this one instead fills in `DEFAULT_SETTINGS`' values for whichever settings are still unset, seeds `EXAMPLE_CATALOG_ITEMS` (`constants.ts`) into the catalog folder (skipping any that already exist), and creates/opens an "Example Trip" folder — a genuine one-click demo rather than "configure things, then click this."

## Localization

- Obsidian has no public i18n API for plugins. Language is detected from `window.localStorage.getItem("language")` — the key Obsidian itself writes when the user sets Settings → General → Language. Unofficial but the established community-plugin pattern.
- All fixed UI strings (settings labels/descriptions, command names, modal titles/buttons/placeholders, notices) go through `t(key, vars?)` from `src/i18n`, keyed by dot-path strings defined in `en.ts` (the type-checked source of truth) with a matching `de.ts`. User data (trip names, category names, catalog item names) is never translated — only the plugin's own chrome.
- Locale is resolved once per plugin load, not reactively — changing the Obsidian language takes effect after reloading the plugin, same as command names elsewhere in Obsidian.
- Adding a language: create `src/i18n/<code>.ts` typed as `Record<keyof typeof en, string>` (a missing key is a type error) and register it in the `LOCALES` map in `src/i18n/index.ts`.

## Key Constraints

- Not desktop-only (`isDesktopOnly: false`) — avoid Electron/Node-only APIs and regex lookbehind (`(?<=`/`(?<!`, unsupported before iOS 16.4) in `src/`
- Pure logic (generation, reset, catalog parsing) belongs in `packing-list.ts` so it stays unit-testable without mocking the Obsidian API
- Text-input suggestions use Obsidian's `AbstractInputSuggest` (`CommaListSuggest` in `main.ts`), not an HTML `<datalist>` — WebKit (iOS/mobile Obsidian) never implemented `<datalist>`'s dropdown, so it silently shows no suggestions there. `minAppVersion` is `1.6.6` to match the `AbstractInputSuggest` API surface in use. `CommaListSuggest` only suggests/replaces the segment after the last comma, so it works for multi-value fields without disturbing already-typed categories.
- Test on mobile with `this.app.emulateMobile(true)` in the desktop dev console before relying on desktop-only testing, per Obsidian's mobile development guide
